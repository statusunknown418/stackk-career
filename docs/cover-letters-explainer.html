Cover Letters: cómo CASEY escribe (y tú editas) una carta

Esto es lo que construimos en /dash/letters/$generationId. No es solo un formulario: es un agente de IA que redacta una carta estructurada, la transmite en vivo sección por sección, la versiona, y te deja editarla a mano o exportarla a PDF. Acá está cada pieza, qué hace y dónde vive.

El recorrido de una carta — de un click al PDF

1

Creas la carta

Pones el puesto + la descripción de la vacante y eliges el idioma (es / en). El form dispara createGeneration y luego trigger.

letters-create-form.tsx → oRPC letters.createGeneration

2

El API arma el thread y lanza el job

Inserta un mensaje-artifact vacío (para que la UI ya muestre la fila) y dispara la tarea de Trigger.dev con concurrencyKey por generación (re-triggers de la misma carta se serializan) e idempotencyKey.

routers/letters.ts → trigger → tasks.trigger(caseyLettersTask)

3

CASEY redacta (en el server, fuera del request)

La tarea corre el agente: lee tu CV en texto plano + la vacante, aplica las reglas duras y los ejemplos, y produce un objeto CoverLetter con Output.object. Modelo: Gemini 3.1 Flash Lite (vía el AI gateway, no Anthropic).

jobs/agents/casey-letters.handler.ts · streamText + tools + Output.object

4

El stream llega en vivo al navegador

La página se suscribe al run con Trigger.dev Realtime. El objeto llega parcial: primero el saludo, luego el cuerpo, el cierre y la firma. Cada sección muestra un shimmer mientras se escribe y un skeleton mientras no empieza.

routes/.../$generationId.tsx (realtime) → letters-artifact-panel.tsx

5

Tú decides qué hacer con ella

Copiar (texto plano), Descargar (PDF con jsPDF), Editar a mano (se guarda sin gastar una versión) o Regenerar con un preset de tono o cambiando el idioma (eso sí crea una versión nueva, hasta el tope).

ArtifactToolbar · updateArtifact (editar) vs trigger (regenerar)

El contrato: qué ES una carta

Todo el feature gira alrededor de este objeto. CASEY lo produce, la DB lo guarda, los dos paneles lo renderizan. Cuatro secciones de texto plano, con topes generosos que cortan ataques de inyección: 

// packages/schemas/src/ai/cover-letter.ts
export const coverLetterSchema = z.object({
  greeting:  z.string().min(1).max(300),   // "Estimada Carolina:"
  body:      z.string().min(1).max(6000),  // 2-4 párrafos, concreto, sin clichés
  closing:   z.string().min(1).max(800),   // CTA suave
  signature: z.string().min(1).max(500),   // "Atentamente, María"
});

El comentario del propio archivo lo dice: "Each section is plain text (markdown allowed); a TipTap-based rich editor can wrap this without a schema change." — por eso reutilizar el editor inline encaja sin tocar el schema.

CASEY — el agente, por dentro

// packages/jobs/src/agents/casey-letters.handler.ts
// Vía el AI gateway = Gemini 3.1 Flash Lite. NO Anthropic.
export const CASEY_LETTERS_MODEL          = "google/gemini-3.1-flash-lite";
export const CASEY_LETTERS_FALLBACK_MODEL = "google/gemini-3.1-flash-lite";

const result = streamText({
  model,
  tools,                                  // el agente puede "razonar" con herramientas…
  stopWhen: stepCountIs(CASEY_MAX_STEPS),    // …pero acotado (3 tools + buffer)
  experimental_output: Output.object({ schema: coverLetterSchema }),
});                                        // la salida final SIEMPRE es el objeto tipado

Las reglas duras (por qué la carta no miente ni se rinde)

R1
Sin cambio de carreraNo te postula a algo que tu CV no respalda.
R2
Cero datos inventadosNo fabrica empresas, métricas ni títulos que no estén en el CV.
R3
Concreto u omitirCita logros reales del CV; si no hay, no rellena con humo.
R4
Anti-clichésProhíbe "apasionado", "team player", etc. (lista compartida + validador).
R5
Multi-idiomaSystem prompt + few-shots por idioma (es / en); el idioma vive en la generación.
R6
Nunca se rehúsaSiempre entrega una carta real; jamás pide "actualiza tu CV".
R7
CV pobre → caso estudianteSi el CV es escaso, lo trata como práctica/primer empleo con lo que SÍ hay.
Defensa anti-inyección.

La vacante y tus notas entran encerradas en fences (<JOB_DESCRIPTION>…</JOB_DESCRIPTION>), así un "ignora las instrucciones anteriores" pegado en la descripción se trata como datos, no como orden. Y los topes del schema + un validador de clichés en runtime detectan si el modelo igual se desvía.

El modelo de datos — un hilo, muchas versiones

generations · el hilo

La carta como conversación

Una fila por carta. Guarda el dueño, el idioma y el resumeId del CV base. Todo lo que ves en /letters/$generationId cuelga de aquí.

messages · las versiones

Cada versión es un mensaje

Un mensaje-artifact por versión, con objectType = COVER_LETTER y el object = CoverLetter. Se ordenan por order → createdAt → rowid (el rowid desempata cuando dos coinciden). 

// "Editar" NO crea versión: sobrescribe el objeto del mensaje mostrado.
updateArtifact(messageId, letter)   // → check de dueño + objectType, sin job ni cuota

// "Regenerar" SÍ crea versión: dispara CASEY otra vez (hasta el tope).
trigger({ extraPrompt, language })   // → nuevo message-artifact en el mismo hilo
