01 · Existen DOS copias del proyecto

Lo que más confunde al inicio. Tu código vive en dos lugares a la vez, y casi todo en git es mover cambios entre ellos.

💻

Tu computadora

"local"
main
feat/letters-casey-i18n-export
… donde editas y commiteas
sincronizas
con ⇅
☁️

GitHub

"remoto" = origin
origin/main
origin/feat/…
… donde el equipo comparte + CI

origin es solo el apodo de "tu GitHub remoto". origin/main = "así está main allá". main a secas = "así está aquí en mi compu". Pueden estar diferentes — por eso existen push y pull.

02 · push vs pull = solo dirección

Una vez que entiendes que hay dos copias, esto es obvio: ¿en qué sentido muevo los cambios?

💻 tu compugit push ▶ subir☁️ GitHub
💻 tu compu◀ git pull · bajar☁️ GitHub

git push

tu compu ──▶ GitHub

Ya hiciste commits locales y quieres que existan en GitHub (para el PR, el equipo, el deploy). Es lo que hicimos con el lockfile: commit → push → apareció en el PR #14.

git pull

GitHub ──▶ tu compu

Alguien subió cambios a GitHub y los traes a tu compu para no quedarte atrás. Usa git pull --rebase para no crear merge-commits basura tipo "Merge X into X".

03 · El historial = cadena de commits

Cada git commit es un punto de guardado: qué cambió, quién, cuándo y un mensaje. El historial es esa cadena enlazada.

A
inicio
B
feat landing
C
feat letters
D
fix lockfile ← HEAD

Cada commit apunta al anterior. Eso es el historial: puedes viajar a cualquier punto, ver el diff o revertir. Como el historial de Google Docs, pero tú decides cuándo se crea cada versión y le pones nombre.

04 · Ramas = líneas paralelas

main es la rama oficial (lo que va a producción). No trabajas directo ahí — creas una rama para experimentar sin romper lo oficial.

main:     A ─► B ──────────────────► M   (estable; recibe el merge al final)
                   ╲               ╱
feat/letters:     C ─► D ─► E   (experimentas tranquilo aquí)

Si rompes algo en feat/letters, main sigue sano. Cuando tu trabajo está listo, traes esos commits (C, D, E) de vuelta a main. Ese "traer de vuelta" se llama merge.

05 · ¿Por qué un Pull Request?

Podrías mergear tu rama a main tú solo con un comando. El PR agrega una puerta de control antes de que algo entre a lo oficial. Un PR es literalmente: "pido que jalen mi rama hacia main".

tu rama
feat/…
commits C,D,E
▶

PULL REQUEST · la puerta

  • Diff — qué líneas cambian
  • Review — alguien comenta/aprueba
  • CI — corren los tests*
  • Registro— queda el porqué
▶
main
oficial ✓

*Por eso importaba arreglar el lockfile: sin eso, el CI del PR fallaba. Cuando apruebas el PR ("Merge"), GitHub fusiona los commits de tu rama dentro de main y recién ahí el cambio es oficial.

06 · Todo junto, en tu flujo real

Los pasos 4 y 8 son los que conectan tu compu ⇄ GitHub: push sube, pull baja.

  1. git checkout -b feat/algocreas una rama (línea paralela)
  2. …editas archivos…el trabajo real
  3. git commitpunto de guardado local en tu compu
  4. git pushsubes la rama a GitHub (origin)
  5. abres un Pull Requestpropones meter esa rama a main
  6. review + CIla "puerta": se valida
  7. Mergelos commits entran a main (oficial)
  8. git checkout main && git pullbajas el main actualizado a tu compu
