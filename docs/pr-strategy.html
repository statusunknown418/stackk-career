2 · La decisión: ¿cómo entra #14 a main?

recomendado Merge commit

Conserva los 15 commits con sus mensajes (feat(letters):…, fix(letters):…). El historial cuenta la historia de cómo se construyó CASEY.

Ideal si valoras la trazabilidad por capa.

Squash

Colapsa todo en 1 commit en main (“feat: CASEY cover letters”). Main queda ultra-limpio pero pierdes el detalle por capa.

Ideal si prefieres 1 línea por feature en main.

3 · Pasos (con comandos)

  1. Re-apuntar #14 a main La punta ya tiene todo el stack, así que su base deja de ser #13 y pasa a ser main. 
    gh pr edit 14 --base main
  2. Cerrar las capas intermedias #9–#13 Su contenido ya viaja dentro de #14; cerrarlas evita 5 merges redundantes. 
    gh pr close 9 10 11 12 13 \
  --comment "Consolidado en #14 (toda la feature CASEY junta)."
  3. (Opcional) Sacar el fix(web) de #14 El bugfix de login/links/sidebar no es de cartas; podría ir en su propio PR a main. Si prefieres velocidad, déjalo dentro. 
    # solo si quieres separarlo — si no, salta este paso
git checkout main && git checkout -b fix/web-bugfix-batch
git cherry-pick d28bfed && git push -u origin HEAD
  4. Mergear #14 a main Elige --merge (conserva historial) o --squash (1 commit) según la sección 2. 
    gh pr merge 14 --merge --delete-branch
# o, para colapsar todo en un commit:
# gh pr merge 14 --squash --delete-branch
  5. Borrar ramas del stack ya cerradas Quedan obsoletas tras cerrar #9–#13. 
    git push origin --delete \
  feat/letters-casey feat/letters-casey-api feat/letters-casey-task \
  feat/letters-casey-polish feat/letters-casey-quality
  6. Limpiar ramas locales muertas Nunca tuvieron PR ni push; son scratch. 
    git branch -D fix/letters-qa-cleanup local/test-with-landing feat/landing
  7. Landing (#7) sigue su propio camino Es independiente de cartas; se revisa y mergea aparte cuando esté listo. 
    gh pr ready 7   # cuando aplique; luego gh pr merge 7 --merge

4 · Reglas para no volver a enredarnos

  1. 1 PR = 1 concernSi el título necesita un “+”, probablemente son 2 PRs. (#14 era i18n + export + hard rules + bugfix.)
  2. git pull --rebase siempreEvita los merge-commits “Merge X into X” que ensucian el historial.
  3. Stack corto y todo ready, o un solo PRSi apilas, mantén ≤3 capas y todas “ready”. Si dudas, una rama de feature directa a main.
  4. Borra la rama al mergear--delete-branch deja el repo sin zombis.
