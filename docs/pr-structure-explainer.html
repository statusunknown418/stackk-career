1 · Qué es un “stacked PR”

En lugar de un PR gigante, partimos la feature de cartas (CASEY) en capas pequeñas. Cada PR no apunta a main: apunta al PR de abajo. Así cada review es chiquito y enfocado. Se mergean de abajo hacia arriba.

6 · Cómo lo ordenamos 🧹

  1. Decidir el destino del stack¿Mergeamos las 6 capas a main tal cual (de abajo hacia arriba), o colapsamos todo en un solo squash-merge a main? Si el review ya está hecho, squash es lo más simple.
  2. Quitar el draft de #9–#11Marcar “Ready for review” la base del stack para que se pueda mergear bottom-up. Hoy están bloqueando.
  3. (Opcional) Sacar el fix(web) de #14El bugfix batch de login/links/sidebar merece su propio PR — no es i18n de cartas.
  4. Borrar branches muertaslocal/test-with-landing, fix/letters-qa-cleanup, feat/landing si ya no se usan.
  5. Adoptar git pull --rebasePara no volver a generar merge-commits de pull que ensucian el historial.
